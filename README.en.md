# SketchFlow

Collects RSS feeds on a schedule and classifies them into three categories:
- Dev Ideas (`개발 아이디어`)
- Dev Issues/Controversies (`개발 이슈/논쟁`)
- New Tools/Products (`새 도구/제품`)

Items in "Dev Ideas" additionally trigger a persona-style pros/cons discussion.
Summaries and discussions are generated by an LLM and stored in both Korean and English.

## Product Intent

- Reduce information overload by clustering items into **ideas / issues / tools**
- Auto-generate pros/cons discussions for **Dev Ideas** to support decisions
- Keep it lightweight and prototype-friendly (JSON storage + minimal FE)

## System Architecture

- **Fetch**: poll RSS feeds via `rss-parser`
- **Classify**: keyword-based categorization (`src/categorize.js`)
- **Summarize/Discuss**: an LLM provider produces KR/EN outputs — selectable via `LLM_PROVIDER`
  - `gemini` — Google Gemini API
  - `lmstudio` — a local, OpenAI-compatible LM Studio endpoint
- **Store**: `data/items.json` (atomic writes, capped at 500 items)
- **Serve**: Express API + static frontend

## Features

- Auto fetch on a configurable schedule (also runs once at startup)
- At most `MAX_ITEMS_PER_FETCH` items are summarized per run, so a large first-run
  backlog does not block for a long time — the rest are picked up on the next tick
- Keyword category classification
- LLM summary + discussion (Gemini or LM Studio)
- Minimal FE: language toggle, summary length toggle, search / sort / source filter, page size
- Expandable cards with discussion collapse/expand (state preserved across re-renders)
- All feed/LLM content is HTML-escaped on render (stored-XSS safe)

## Install

```bash
npm install
```

## Environment

Create `.env` based on `.env.example`.

```env
PORT=3000
FETCH_EVERY_MINUTES=60         # prefer a divisor of 60 (e.g. 15, 30, 60)
MAX_ITEMS_PER_FETCH=25         # items summarized per fetch run
FETCH_TOKEN=                   # if set, POST /fetch requires header x-fetch-token

# LLM_PROVIDER selects the summarizer: "gemini" (default) or "lmstudio"
LLM_PROVIDER=gemini

# Gemini (LLM_PROVIDER=gemini). Without a key the server still runs and fills
# summaries with a placeholder message.
GEMINI_API_KEY=your_api_key_here
GEMINI_MODEL=gemini-1.5-flash

# LM Studio (LLM_PROVIDER=lmstudio) — a local OpenAI-compatible server
LMSTUDIO_URL=http://localhost:1234/v1
LMSTUDIO_MODEL=qwen2.5-32b-instruct
LMSTUDIO_TEMPERATURE=0.3
LMSTUDIO_MAX_TOKENS=500
LMSTUDIO_MAX_CONTEXT_LENGTH=8000
LMSTUDIO_TIMEOUT=120000
```

> Scheduling note: `FETCH_EVERY_MINUTES` is turned into a cron expression. Values
> below 60 use a minute step (`*/N * * * *`); whole-hour values use the hour field;
> other values above an hour are rounded to the nearest hour.

## API

- `GET /health`
- `GET /feeds`
- `GET /items` — all items
- `GET /items?category=개발%20아이디어` — filter by category (values are the three
  Korean category strings above; the filter is an exact match)
- `POST /fetch` — trigger a fetch now (requires `x-fetch-token` if `FETCH_TOKEN` is set)

## Frontend

Open `http://localhost:3000` after the server starts.

## Data Storage

- `data/items.json` — live data (git-ignored)
- `data/items.sample.json` — a committed sample showing the item shape

## Notes

- Live `data/` files are excluded from commits; only `data/items.sample.json` is tracked.
- If existing data lacks summary/discussion fields, re-fetch to populate them.
- When `FETCH_TOKEN` is set, the web UI's "fetch now" button prompts for the token
  once and remembers it in the browser (`localStorage`).
- This is a single-operator local dashboard. It has no authentication beyond the
  optional `FETCH_TOKEN`; do not expose it to untrusted networks as-is.

## Roadmap & Future Directions

This repository is a working prototype — deliberately small, file-backed, and
single-operator. Think of it as a foundation to build on rather than a finished
product. Some directions worth exploring:

- **Smarter classification.** The categorizer is keyword-based and Korean/English
  only. Embedding- or LLM-based classification (with confidence scores and
  user-defined categories) would handle nuance and new topics far better.
- **A real datastore.** `data/items.json` is easy to reason about but caps at 500
  items with no indexing. SQLite/Postgres would unlock full-text search, dedup on
  normalized URLs, retention policies, and safe concurrent access.
- **A more robust LLM pipeline.** Calls are currently sequential. Bounded
  concurrency, retries with backoff, response caching, and stricter output
  validation/sanitization (defense-in-depth against prompt injection) would make
  ingestion faster and safer. Embeddings could also cluster related items.
- **User-managed feeds.** Feeds are hard-coded in `src/feeds.js`. Adding/removing
  feeds from the UI (with OPML import and per-feed rules) would make it reusable.
- **A richer frontend.** The vanilla-JS FE is intentionally minimal. Read/unread
  state, favorites, live updates (SSE/WebSocket), and theming are natural next
  steps; a small framework would help if it grows.
- **Production readiness.** Real authentication and multi-user support, rate
  limiting, containerization, and CI (running `npm test` + `npm audit` on every
  push) would take it from "runs on my machine" to "deployable."

Contributions and forks are welcome — treat the current code as a starting point,
not a constraint.
